VclRotatedEdit is a single-line VCL edit control that can render and edit text
horizontally, vertically, or at an arbitrary angle while preserving the behavior
expected from an edit control:

  - caret movement;
  - selection;
  - clipboard operations;
  - validation;
  - VCL styles;
  - design-time resizing;
  - native-like logical AutoSize behavior;
  - style-aware border metrics;
  - backend separation for future Direct2D/DirectWrite rendering.

MAIN CONCEPTS
-------------

The component separates the normal VCL external bounds from the logical edit
surface used internally by the editor.

  Left, Top, Width, Height
      Normal external VCL rectangle.

  LogicalLength
      Logical size along the text flow axis.

  LogicalThickness
      Logical size perpendicular to the text flow axis.

  Angle
      Effective rotation angle in degrees.

  Orientation
      Convenience property for common modes:
      horizontal, vertical down, vertical up and custom angle.


VALIDATED ORIENTATION CONVENTION
--------------------------------

  reoVerticalDown
      Visual text flow starts near the top of the control and goes downward.
      This orientation maps to 270 degrees.

  reoVerticalUp
      Visual text flow starts near the bottom of the control and goes upward.
      This orientation maps to 90 degrees.

This convention must remain centralized in the geometry unit so that rendering,
caret placement, selection, hit-testing and layout all use the same angle.


AUTOSIZE AND LOGICAL THICKNESS
------------------------------

AutoSize uses the familiar TEdit name, but it updates LogicalThickness rather
than treating Width and Height as the edit surface.

When AutoSize is enabled, LogicalThickness is resolved from a native TEdit probe
created in the same parent/style context. This keeps the logical edit height
close to the current VCL style, font and border settings.

When AutoSize is disabled and LogicalThickness is reduced below the preferred
value, the layout keeps the normal top margin and lets the lower part clip, as
a native single-line edit does. When LogicalThickness is enlarged, the visual
text band is centered in the enlarged editable area.

DESIGN-TIME BEHAVIOR
--------------------

The component intentionally separates the external VCL rectangle from the
internal editable surface.

Important rules:

  - Loading a DFM preserves the streamed external bounds.

  - Changing LogicalLength or LogicalThickness recalculates the external bounds
    from the logical edit surface.

  - Changing Angle or Orientation recalculates the external bounds while keeping
    the external center stable.

  - During a design-time resize drag, only one logical axis changes:
    either LogicalLength or LogicalThickness, never both.

  - If a design-time resize would produce an invalid or too-small rectangle, the
    component restores the last valid geometry.

  - During design-time dragging, the control avoids repainting the full parent
    surface into its own background. This prevents containers such as TGroupBox
    from leaking their border or caption into the rotated edit background.


STYLE SUPPORT
-------------

The component is owner-drawn and resolves its colors through PaletteMode and
StyleElements.

  repmStyle
      Uses the resolved VCL style palette when possible.

  repmCustom
      Uses component colors such as Color, Font.Color and BorderColor.

StyleElements is honored:

  seClient
      Style controls the edit background.

  seFont
      Style controls the text color.

  seBorder
      Style controls the border color.


RENDERING BACKEND BOUNDARY
--------------------------

Rendering and text-metric dependent operations pass through
IRotatedEditRenderBackend. The component currently provides two backends:

  rebDirect2D
      Default Direct2D / DirectWrite renderer used when native resources are
      available. It draws in the final oriented coordinate system.

  rebGDI
      Historical renderer kept as an explicit compatibility backend and safety
      fallback.

The backend contract includes layout and hit-testing, not only drawing, because
editable text requires drawing, caret placement, selection geometry and
hit-testing to stay metrically coherent.

DIRECT2D ARCHITECTURE RULE
--------------------------

The Direct2D backend draws in the final oriented coordinate system of the
control. It must not draw a straight edit into a bitmap and rotate or project
that bitmap afterwards. If a double buffer is needed, the buffer must already
represent the final oriented control surface.

This rule avoids the edge artefacts that can appear when already-rasterized GDI
content is projected to a shallow angle.

CURRENT RENDERING STATE
-----------------------

The GDI backend remains available and is used as a fallback if Direct2D /
DirectWrite resources cannot be created or if a Direct2D drawing operation
fails.

The Direct2D backend owns the Direct2D factory, DirectWrite factory and DC render
target. It draws the background, frame, text, selection and caret. DirectWrite is
used for native text positions used by selection and caret placement.

The selection is logically preserved when the control loses focus, but the visual
highlight is hidden while the control is not focused.
